# MQcafe 독립형 무인 스터디카페 시스템 체크리스트 및 구조 평가서

본 문서는 N100 미니 PC를 기반으로 로컬망에서 작동하도록 구축된 MQcafe(스터디카페) 시스템의 핵심 구성 요소를 점검하고, 각 기술적/비즈니스적 체크포인트에 대한 **논리적 타당성 및 현실적 한계(잠재적 오류 가능성)**를 평가합니다.

---

## 1. 하드웨어 및 인프라 (N100 로컬 서버)

- [x] N100 미니 PC에 로컬 서버(API, MQTT, DB) 호스팅 환경 구축
- [x] 인터넷 장애 시에도 매장 내(로컬망) 서비스 유지 가능 여부
- [x] 외부 접근을 위한 포트포워딩 또는 Ngrok/Cloudflare Tunnel 적용 계획

> **📊 구조 평가 (Practicality & Logic)**
> - **논리성 (우수):** 외부 클라우드 의존도(Supabase 등)를 낮추고 매장 자체 서버를 돌리는 것은 인터넷 단절에도 오프라인 출입 통제(ESP32 NFC)를 가능하게 하므로 무인 매장에 매우 논리적이고 훌륭한 선택입니다.
> - **현실적 한계/오류 가능성 (주의):** 미니 PC의 전원이 끊기거나(정전), 로컬 공유기가 다운되면 전체 시스템이 멈춥니다. **정전 대비 UPS(무정전 전원 장치) 도입**이나 주기적인 외부 클라우드 백업(하루 1회)이 필수적입니다. 또한 고정 IP가 없는 환경에서는 원격 관리를 위한 DDNS 설정이 반드시 필요합니다.

---

## 2. 데이터베이스 및 저장소 (SQLite JSONB 기반)

- [x] SQLite를 이용한 `table_sessions`, `nfc_cards` 등 스키마 구축
- [x] JSONB 컬럼을 활용한 메타데이터(채팅, 결제 정보 등) 비정규화 저장
- [x] Connection Pooling (`psycopg2.pool` 또는 SQLite 쓰레드 제한 해결) 적용 여부

> **📊 구조 평가 (Practicality & Logic)**
> - **논리성 (양호):** 미니 PC 스펙에 맞게 가벼운 SQLite(또는 PostgreSQL 로컬 설치)를 사용한 점과, 유연한 확장을 위해 JSONB에 메타데이터를 담은 것은 초기 스타트업 및 로컬 서비스 개발에 매우 빠르고 효율적인 접근입니다.
> - **현실적 한계/오류 가능성 (주의):** SQLite 사용 시 동시 쓰기(Write) 작업이 많아지면 Database Locked(DB 잠금) 에러가 발생할 수 있습니다. 현재 Connection Pool을 도입하여 완화했으나, 향후 결제와 출입 스캔이 동시에 수십 건씩 몰리는 대형 매장에서는 **로컬 PostgreSQL**로 마이그레이션 하는 것이 물리적으로 훨씬 안정적입니다. (현재 구조 상 PostgreSQL 전환은 매우 쉽습니다.)

---

## 3. 백엔드 및 실시간 통신 (FastAPI, MQTT, WebSocket)

- [x] FastAPI를 통한 RESTful API 라우터 (세션, 결제, 점주 로직) 모듈화 분리
- [x] ESP32 도어락과의 실시간 통신을 위한 로컬 MQTT 브로커(Mosquitto 등) 연동
- [x] 점주와 고객 간 실시간 채팅을 위한 WebSocket 통신 구현

> **📊 구조 평가 (Practicality & Logic)**
> - **논리성 (우수):** IoT 하드웨어(도어락, 키오스크 단말기) 제어는 HTTP보다 가볍고 빠른 MQTT를 활용하고, 웹 UI(채팅 등)는 WebSocket을 혼용한 하이브리드 통신 방식은 매우 현대적이고 정확한 아키텍처 설계입니다.
> - **현실적 한계/오류 가능성 (양호):** MQTT Broker가 미니 PC에서 죽으면 출입문이 열리지 않는 치명적 에러가 발생할 수 있습니다. 백엔드 서버(FastAPI)와 MQTT 브로커 서비스(예: Mosquitto)가 서버 부팅 시 즉시 자동 실행되도록 OS 단의 서비스 등록(Systemd 또는 Windows Services)이 확실하게 세팅되어야 합니다.

---

## 4. 프론트엔드 UI/UX (React + Vite + Lazy Loading)

- [x] 고객 모드(키오스크), 점주 모드(대시보드), 관리자 모드 통합 구현
- [x] `React.lazy`를 통한 코드 스플릿팅(번들 최적화)
- [x] 실시간 좌석 맵(Map) 및 애니메이션 적용, 반응형 레이아웃 도입

> **📊 구조 평가 (Practicality & Logic)**
> - **논리성 (우수):** 하나의 앱(App.tsx)에서 라우팅 또는 모드 변경으로 다양한 뷰를 제공하면서도, Lazy Loading을 통해 불필요한 코드를 분리한 것은 퍼포먼스 측면에서 매우 효율적입니다.
> - **현실적 한계/오류 가능성 (양호):** 키오스크 모니터(터치스크린)에서 구동될 고객 모드의 경우, 브라우저의 새로고침이나 뒤로 가기 제스처 등에 취약할 수 있습니다. 실제 상용화 시에는 단순 웹 브라우저가 아닌 **Kiosk Mode(전체화면, 크롬 제어 차단 모드)**로 강제 실행하는 스크립트나 앱 래퍼(Electron 등)로 패키징해야 사용자의 오작동을 막을 수 있습니다.

---

## 5. 비즈니스 로직 (세션 및 출입 관리)

- [x] 결제(예약) -> 5분 내 출입 인증(체크인) -> 외출/복귀 토글 -> 퇴실 플로우 구현
- [x] 만 16세 미만 출입 제한 로직 (생년월일 필터)
- [x] 이용 시간 종료 후 자동 퇴실(Cron Archive) 스케줄링 구현

> **📊 구조 평가 (Practicality & Logic)**
> - **논리성 (우수):** 예약 후 5분 내 미입장 처리, 외출 중 시간 차감 등의 세밀한 룰은 한국 무인 스터디카페의 운영 실태를 완벽하게 반영한 논리적 구조입니다.
> - **현실적 한계/오류 가능성 (주의):** 
>   1. **인증 우회:** 현재 '실명인증' 버튼이 모의(Mocking) 처리되어 있습니다. 악의적인 미성년자가 임의의 성인 생년월일을 입력하면 우회할 수 있습니다. 상용 배포 전 KCB나 PASS와 같은 실제 실명/본인인증 API 연동이 필요합니다.
>   2. **자동 퇴실 한계:** 사용자가 실제로 짐을 두고 나갔는지, 완전히 퇴실했는지 시스템은 100% 알 수 없습니다. 만료 시간이 되어 자동 퇴실 처리되었는데 짐이 그대로 남아있는(사석화) 문제에 대해 점주에게 알람을 주거나, 강제 퇴실 처리를 할 수 있는 점주용 룰이 강화되어야 합니다.

## 6. 결제 및 단말기 연동 (NicePay / Terminal)

- [x] NicePay 등 PG사 연동을 위한 백엔드 인터페이스 구축
- [x] 오프라인 단말기 하트비트(Heartbeat) 수신 및 결제 대기 흐름 구축
- [x] 결제 성공 여부에 따른 UI 분기 처리 완료

> **📊 구조 평가 (Practicality & Logic)**
> - **논리성 (양호):** MQTT를 활용하여 오프라인 카드 단말기에 결제 요청(pay_request)을 보내고 결과(pay_result)를 비동기로 받는 아키텍처는 스마트 자판기나 무인 매장에서 쓰이는 표준적이고 안정적인 방식입니다.
> - **현실적 한계/오류 가능성 (주의):** 결제를 요청한 상태에서 고객이 카드를 꽂지 않고 화면을 닫아버리거나 이탈할 경우(네트워크 지연 포함), 단말기에는 결제 대기 창이 계속 떠 있을 수 있습니다. 따라서 **결제 타임아웃 롤백** 로직(ex: 60초간 응답 없으면 단말기에 결제 취소 명령 전송)이 꼼꼼하게 처리되어야 현실적인 장애를 예방할 수 있습니다.

---

### 💡 총평 (Overall Conclusion)

현재 구축된 **MQcafe 시스템의 아키텍처(N100 로컬 서버 + MQTT/WS 실시간 연동 + Vite 기반 통합 프론트엔드)**는 클라우드 구독 비용을 없애고 안정성을 취하는 무인 오프라인 매장의 목적에 **현실적으로 매우 부합하며 훌륭하게 설계**되었습니다.

다만, 상용화(실제 매장 도입) 직전에 마주할 수 있는 하드웨어적 예외 상황(단말기 타임아웃, 미니PC 전원 차단, 악의적 가짜 생년월일 기입)에 대한 **물리적 예외 처리와 보안 인프라 보강**만 거친다면 즉시 시장에 투입 가능한 수준의 완성도입니다.
